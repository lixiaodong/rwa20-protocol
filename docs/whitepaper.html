1. Abstract

RWA20 is an open-source, permissionless smart contract protocol designed to become the universal standard for tokenizing real-world assets (RWA) on Ethereum and EVM-compatible blockchains. The protocol addresses the fundamental bottleneck in RWA adoption: compliance.

By embedding multi-jurisdiction legal compliance directly into the ERC20 token transfer mechanism, RWA20 ensures that compliance cannot be bypassed by any transfer route — including decentralized exchanges, bridges, or direct wallet transfers. The protocol supports six initial jurisdictions (Singapore, Switzerland, UAE, Liechtenstein, Malaysia, United States) and is architected for unlimited extensibility.

Core thesis: Real-world asset tokenization has failed to scale not because the technology is immature, but because compliance has been treated as an add-on rather than a protocol primitive. RWA20 makes compliance a first-class citizen at the token layer.

2. Background & Problem Statement

2.1 The $500 Trillion Opportunity

Global real-world assets — real estate, private equity, bonds, commodities, infrastructure — represent approximately $500 trillion in value. DeFi's total locked value peaked at approximately $180 billion. The gap is not technological: it is legal, regulatory, and structural.

2.2 Why Existing Approaches Fail

Current RWA tokenization platforms share four fundamental limitations:

  • Compliance as an external oracle: Most platforms check compliance via off-chain KYC providers or permissioned transfer restrictions. These are bypassable via DEX liquidity pools or direct on-chain transfers.
  • Single-jurisdiction architecture: A token that satisfies US Reg D cannot easily satisfy Singapore MAS §274 or Swiss FINMA requirements simultaneously. Issuers must choose one market.
  • No on-chain legal proof: Token holders cannot prove on-chain that their token corresponds to a specific legal document, court record, or regulatory filing. The link exists only off-chain.
  • Closed platforms: Each platform (Centrifuge, Maple, Ondo, Securitize) builds a proprietary compliance stack. There is no interoperability, no standard, and no composability between platforms.

2.3 The Standard Gap

The Ethereum ecosystem has powerful standards for fungible tokens (ERC20), non-fungible tokens (ERC721), and semi-fungible tokens (ERC1155). For security tokens, EIP-3643 (T-REX) provides a compliance framework focused on on-chain identity. However, no existing standard provides: (a) multi-jurisdiction compliance as a native protocol primitive, (b) on-chain legal attestations linking tokens to real-world legal records, and (c) an integrated economic model (launchpad + revenue distribution) suitable for any asset class. RWA20 fills this gap.

3. Protocol Design Principles

  1. Compliance as primitive: Legal compliance must be enforced at the lowest level of the token implementation, not as an external layer that can be bypassed.
  2. Jurisdiction agnosticism: The core protocol must be neutral to specific legal frameworks. Jurisdictions are plug-in modules.
  3. Extensibility over completeness: It is better to ship a great extension mechanism than to pre-build every jurisdiction. A well-designed standard grows by contribution.
  4. On-chain legal provenance: The link between a token and its underlying legal document must be verifiable on-chain — not just via the issuer's word.
  5. Economic neutrality: The protocol must not impose rent-seeking fees that disadvantage any issuers or investors. Start at zero; earn trust first.
  6. Open source as strategy: The goal is not to be the biggest platform. The goal is to be the standard. The ERC20 contract has no business model — yet the entire DeFi ecosystem runs on it.

4. Technical Architecture

4.1 Core Token Contract

The RWA20Token contract inherits from OpenZeppelin v5's ERC20, ERC20Votes, ERC20Permit, AccessControl, and Pausable. The compliance check is placed inside the internal _update() function:

function _update(address from, address to, uint256 value) internal override {
    if (from != address(0) && to != address(0)) {
        require(
            compliance.canTransfer(from, to, value),
            "RWA20: transfer blocked by compliance"
        );
    }
    super._update(from, to, value);
    // Auto-delegate to self on first receive for governance snapshots
    if (to != address(0) && delegates(to) == address(0)) {
        _delegate(to, to);
    }
}

4.2 Compliance Architecture

The compliance layer follows a strict interface hierarchy:

4.3 Multi-Jurisdiction OR Logic

The MultiJurisdictionCompliance contract applies OR-logic across enabled jurisdictions. A transfer is permitted if and only if both parties each satisfy at least one enabled jurisdiction:

function canTransfer(address from, address to, uint256 value)
    external view returns (bool) {
    bool fromQualifies = false;
    bool toQualifies = false;
    for (uint i = 0; i < enabledCodes.length; i++) {
        address mod = registry.getModule(enabledCodes[i]);
        if (!fromQualifies && IJurisdictionModule(mod).isActiveInvestor(from))
            fromQualifies = true;
        if (!toQualifies && IJurisdictionModule(mod).isActiveInvestor(to))
            toQualifies = true;
        if (fromQualifies && toQualifies) return true;
    }
    return false;
}

The RevenueDistributor uses ERC20Votes snapshots for O(1) per-holder claims, avoiding the need to iterate over all holders:

5.2 Jurisdiction-Specific Rules

Each jurisdiction module enforces its local regulatory requirements. Key examples:

  • Singapore: Max 50 investors for private placement (SFA §275 threshold). Auto-rejects if registeredCount >= 50.
  • Switzerland: Lock-up period tracking. CHJurisdiction.onTransfer() records firstPurchaseAt and blocks early transfers.
  • UAE: On-chain sanctions enforcement. Countries loaded into _sanctionedCountries mapping; investor country checked at registration.
  • Liechtenstein: FMA registration number stored on-chain. Prospectus approval determines minimum investor class (RETAIL vs SOPHISTICATED).
  • US: Reg D 506(b) enforces max 35 non-accredited investor cap. Reg S tracks 40-day restricted period. Rule 144 enforces 12-month lockup for restricted securities.

  • Ethereum Attestation Service (EAS): Mainnet address 0xA1207F3BBa224E2c9c3c6D5aF63D0eb1582Ce587. Schema-based attestations with built-in revocation.
  • Kleros: Decentralized court providing dispute resolution and evidence verification. Awards are bridged into the RWA20 attestation registry.

7.2 Fee Routing

Collected fees flow through ProtocolFeeManager.sweep() and are distributed to three pools:

  • Treasury (70%): Protocol development and operations fund
  • Ecosystem (20%): Grants for jurisdiction module contributors, auditors, legal researchers
  • Stakers (10%): RWAG token stakers who participate in protocol governance

8. Governance

Phase 0–1 governance is controlled by the founding team via a Gnosis Safe multisig. Phase 2 transitions to RWAG token governance:

  • 100 million RWAG total supply
  • On-chain proposals via Governor contract (inherited from OpenZeppelin)
  • Parameters under governance control: fee levels (within hard caps), jurisdiction approvals, treasury allocation, grant programs
  • Parameters excluded from governance: hard fee caps, MIT license, core protocol interfaces

9. Security Considerations

  • Compliance bypass prevention: Placement inside _update() ensures no transfer mechanism — including transferFrom(), DEX router calls, or flash loans — can bypass compliance checks.
  • Attestation trust model: Attestors require ATTESTOR_ROLE in the registry. Trust is jurisdiction-scoped: a Singapore attestor cannot create UAE attestations.
  • Fee manipulation: Hard caps are set in contract constructor as immutable constants. No governance action can exceed them.
  • Oracle manipulation: Price feeds use Chainlink's decentralized oracle network with heartbeat checks and price deviation bounds.
  • Snapshot manipulation: Revenue snapshots use block.number - 1 to prevent same-block flash loan attacks on distribution claims.

A formal security audit is planned for Phase 1. Contributions to the test suite and audit preparation are welcome.

11. Conclusion

RWA20 represents a fundamental shift in how real-world asset tokenization is approached: from fragmented platform solutions to a unified, permissionless protocol standard. By making compliance a first-class protocol primitive — not an afterthought — RWA20 enables any asset, in any jurisdiction, to be tokenized with the full force of local law behind it.

The protocol is designed to grow by community contribution. Every lawyer who reviews a jurisdiction module, every developer who builds on the protocol, and every regulator who engages with the standard makes the entire ecosystem stronger. This is the network effect that turns a good protocol into a lasting standard.

We invite developers, legal researchers, asset managers, and regulators worldwide to join us in building the infrastructure that will finally bring the $500 trillion real-world asset market on-chain.
